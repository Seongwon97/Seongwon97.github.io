1. REST API가 아니면 컨트롤러와 뷰를 분리하자

  • 피드백 링크

Spring이 제공하는 controller관련 어노테이션은 @Controller와 @RestController가 있다. 둘의 차이점을 크게 설명하면 RestController는 Json타입의 ResponseBody를 반환하고 Controller는 String타입 또는 ResponseBody를 반환한다는 차이점이 있다. 즉, RestController는 Controller + ResponseBody라고 봐도된다.

둘의 내부 동작 원리를 구체적으로 살펴보면 Controller의 경우 반환값이 String일 경우, ViewResolver가 작동하여 해당 문자열을 가진 View가 있는지 찾아본 후 있으면 해당 View를 반환해주고 그 외의 Object타입일 경우 MessageConverter가 반환값을 변환해줍니다. 반면에 RestController는 Object(Json)타입만을 반환하기에 ViewResolver가 아닌 MessageConverter만 동작하게 됩니다.

결론은 둘의 동작 원리가 다르기에 프로그래밍을 할 때는 @Controller, @RestController가 달린 컨트롤러 클래스를 두개 생성한 후, view와 관련된 메서드는 @Controller가 달린 클래스에 위치시키고 다른 REST API의 메서드들은 @RestController에 위치시키는 것이 좋다는 것이다.

  • 참고 링크- @Controller와 @RestController 차이

2. REST API에서 URI는 자원을 표현하는데 중점을 두고 행위에 대한 표현은 피하자

  • 피드백 링크1
  • 피드백 링크2
  • 피드백 링크3

REST API의 규칙으로는 아래와 같은 내용이 있다.

  1. URI는 정보의 자원을 표현해야 한다. (리소스명은 동사보다는 명사를 사용)
  2. 자원에 대한 행위는 HTTP Method(GET, POST, PUT, DELETE)로 표현

이번 미션을 처음 진행하며 만든 uri의 경우 @PostMapping("/new-game"), @DeleteMapping("/delete") 와 같이 uri내에서 행위를 표현하는 정보가 들어있었다. 이러한 uri는 REST API에는 좋지 못하며 @PostMapping("/game") , @DeleteMapping("/game") 와 같이 자원만을 표현하는 것이 좋다.

4. ResponseEntity의 로(raw) 타입의 사용은 지양하자, void도 타입이다.

  • 피드백 링크

해당 내용은 브리의 체스 미션 피드백에서 어노테이션 관련 불편한 코드의 내용에도 소개된 내용이다. 로 타입의 경우, 잘못된 사용을 하였을 때 컴파일 에러가 발생하지 않고 런타임시에 발생하는 문제가 있습니다. 그래서 로 타입의 사용을 지양해야한다.

사실 이번 미션 코드에서 로 타입을 쓰게 된 이유는 Delete와 같이 반환 값이 없는 요청의 경우 ResponseEntity의 타입을 어떻게 지정해야할지 몰랐기 때문이었다. 럿고의 피드백을 받으며 void도 하나의 타입이라는 것을 알게 되었고 delete와 같이 반환값이 없는 경우 ResponseEntity의 타입을 void로 지정하면 된다는 것을 알게 되었다.

  • 참고 링크 - Void Type in Java

5. 세션의 사용 (feat.페이지의 이동)

  • 피드백 링크

미션을 진행하며 방 생성과 동시에 게임 방으로 접속하는 기능을 구현할 때 아래와 같이 3가지 방법을 고민하였고, 최종적으로 세션을 사용하는 방법으로 미션을 진행하였다.

  1. 페이지 이동을 할 때 Js 단에서 roomId를 넘겨 /game/load/{roomId} get request를 통해 말을 가져오는 방법
  2. 페이지 이동 후, js단에서 url을 파싱하여 roomId를 추출한 후 다시 request를 하여 말을 가져오는 방법
  3. 뷰를 반환하는 /game/{roomId} get request를 할 때, 세션을 저장하고 그 후에 말 정보를 가져오는 /game/load에서 세션에 저장된 방 정보를 통해 말을 가져오는 방법

하지만 세션을 사용하는 것은 REST API의 Stateless한 속성이 지켜지지 않아 좋지 않은 방법이라는 것을 알게 되었다.

  • 참고 링크 - REST API 제대로 알고 사용하기

6. 페이지 이동 방법

위의 피드백에서 진행한 고민에서 이어지는 내용이다. 해당 내용에 대해서 럿고와 줌을 통해 여러 내용을 들으며 해당 내용에 대한 고민이 해결되었다.

요즘 웹 개발에서 페이지 이동의 경우, 보통 프론트 단에서 처리를 해준다고 한다. 그래서 백엔드쪽에서는 보통 데이터 요청에 대한 반환을 하고, 해당 내용을 랜더링하고 페이지를 이동하는 것은 프론트단에서 진행하여 내가 고민했던 내용에 대해서는 부담갖지 않아도 된다고 했다.

그리고 한번의 요청으로 얻을 수 있는 데이터를 여러번의 요청으로 나눠서 데이터를 받아오는 것에 대한 낭비에 대해서도 고민을 하였는데, 해당 내용도 럿고가 실제 서비스의 사례들을 보여주며 실제 서비스들도 한번의 클릭으로도 여러번의 요청을 보낸다는 것을 보여주고, 요즘은 하드웨어의 성능이 좋아져 어느정도의 요청은 괜찮다고 했다.

  • 피드백 링크

일반적으로 save는 insert, update의 내용을 모두 내포하고 있어 둘 다 될 때 사용한다. 단순히 저장 기능을 수행하는 메서드의 경우 insert가 적합하다.

8. DAO 테스트에서는 @JdbcTest를 사용하자

  • 피드백 링크

DAO테스트에서도 controller테스트처럼 @SpringBootTest를 사용할 경우, 해당 테스트에 필요없는 객체들의 빈들도 생성되어 낭비가 발생합니다. 그래서 DAO테스트의 경우 필요한 빈(테스트를 위한 JdbcTemplate)만 생성하여 테스트하는 @JdbcTest를 사용하는 것이 좋다.

  • 참고 링크 - @JdbcTest, @DataJdbcTest, @JpaTest
